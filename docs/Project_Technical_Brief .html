Project Objectives

The primary objective of this project is to build a fully automated, open-source security platform targeted at small enterprises that cannot afford commercial SIEM solutions like Splunk or Microsoft Sentinel. The system collects security events, analyzes them using a locally hosted AI model, makes threat decisions, and executes automated response actions — without requiring a human analyst for routine threat handling.

OBJ-01
Automate Threat Detection
Eliminate manual log review for common attack patterns such as brute force, port scanning, and file integrity violations through rule-based and AI-driven detection.
OBJ-02
AI-Powered Threat Classification
Use a cybersecurity-specialized large language model (Foundation-Sec-8B) to classify threats with MITRE ATT&CK mapping, confidence scores, and severity ratings.
OBJ-03
Automated Threat Response
Block attacking IPs at the firewall level within seconds of detection using automated iptables rules via a custom Python API.
OBJ-04
Host Isolation Capability
Automatically quarantine compromised machines from the network for severe threats like ransomware and C2 beacons, preserving the Wazuh manager connection.
OBJ-05
Real-Time Analyst Notification
Keep the human analyst informed via Telegram instant messages with full alert details — no need to be physically at a workstation.
OBJ-06
Zero Licensing Cost
Provide a fully open-source alternative to commercial SIEM/SOAR platforms. Every component is free and runs entirely on-premise with no cloud dependency.
OBJ-07
Small Enterprise Focus
Deployable on commodity hardware. No expensive cloud infrastructure. Designed for organizations that lack dedicated security teams and budgets.
OBJ-08
Simplified Agent Onboarding
Enable non-technical analysts to onboard new machines into the security platform via an AI-guided onboarding assistant web page.

How the System Works

When an attacker performs a malicious activity such as SSH brute force or port scanning against a monitored machine, the following automated pipeline executes end-to-end within 3 to 6 seconds:

🖥️ Attacker Action
→
🔍 Wazuh Agent Detects
→
📡 Wazuh Manager Alert
→
🔗 custom-webhook.py
→
⚙️ n8n Pipeline
→
🧠 Foundation-Sec-8B AI
→
⚖️ Decision Engine
→
🚫 Block IP / 📱 Telegram / 🔒 Isolate
AI Decision Outputs — Every Alert Gets All 6
Output FieldDescriptionExample
DecisionPrimary automated action to takeMONITOR / INVESTIGATE / BLOCK_IP / ISOLATE_HOST
Threat TypeClassification of the attackSSH Brute Force, Port Scan, Ransomware, C2 Beacon
MITRE ATT&CK IDIndustry-standard attack framework mappingT1110 (Brute Force), T1046 (Network Scan)
Confidence ScoreAI certainty level0–100%
SeverityThreat severity ratingLow / Medium / High / Critical
ReasonHuman-readable explanation of decisionNatural language summary from AI
effectiveCount Auto-Block Logic

The AI receives a 24-hour alert count plus a 5-minute velocity bonus for the attacking IP. If the combined effectiveCount reaches 15 or above, the system automatically escalates to BLOCK_IP regardless of individual alert severity — preventing slow, low-and-slow attack patterns from evading detection.

01
🛡️
Wazuh — SIEM + EDR Engine
CORE
▶
What It Is
Open-source Security Information and Event Management platform with built-in Endpoint Detection and Response capabilities. The core security monitoring engine of the entire platform.
Deployment
Docker containers, single-node setup on Ubuntu SOC server. Version: Wazuh 4.14.2. Config at /var/ossec/etc/ossec.conf inside the container.
Key Features Used
  • Agent-based log collection from endpoint machines
  • Rule-based alert generation (brute force, port scan, suspicious process)
  • File Integrity Monitoring (FIM) on /home/soc and /etc/iptables/
  • Integratord — forwards alerts to n8n via custom webhook script
  • Active Response — triggers isolation scripts on monitored agents
Agents Monitored
  • SIEM-SERVER — the SOC server itself (Agent ID 001)
  • LAB-SERVER — secondary Ubuntu 24.04 VM (Agent ID 003)
Note: Wazuh persistence across container restarts is maintained via a Docker volume (wazuh_integrations) and a systemd service (wazuh-integrations-fix.service) that auto-corrects file ownership on boot.
02
⚙️
n8n — Workflow Automation / SOAR
ORCHESTRATOR
▶
What It Is
Open-source workflow automation platform used as a lightweight SOAR (Security Orchestration, Automation and Response) engine. The central brain that orchestrates all pipeline actions.
Deployment
Docker container, port 5678, on the Ubuntu SOC server. Version: n8n v12.9.4. Pipeline: 42 nodes. maxConcurrency set to 1 to prevent race conditions.
Key Nodes Used
  • Webhook node — receives Wazuh alerts as JSON
  • HTTP Request nodes — calls AI, IP blocker, and Telegram APIs
  • Code nodes (JavaScript) — enriches context, builds AI prompts
  • IF/Switch nodes — routes decisions to action branches
  • InfluxDB nodes — reads history, writes decisions
Network Bridges
n8n bridges both soc_network (172.x.x.x/16) and single-node_default (172.x.x.x/16) Docker networks to reach all internal services.
03
🧠
Foundation-Sec-8B via Ollama — AI Threat Analysis
AI ENGINE
▶
What It Is
Foundation-Sec-8B is an 8 billion parameter Large Language Model specifically trained and fine-tuned by Fortinet on cybersecurity data. It runs locally via Ollama — an open-source local LLM runtime. No data leaves the network.
Model Specs
Foundation-Sec-8B-Instruct, Q4_K_M quantized (~5GB). Runs on Windows host via Ollama on RTX 5060 8GB VRAM (CUDA-accelerated). Accessible to Ubuntu VM over Tailscale at port 11434.
Why Local AI
  • No security data leaves the network
  • No API costs or internet dependency
  • Cybersecurity-specialized — not a generic LLM
  • Suitable for enterprise on-premise deployment
AI Output Fields
Decision, Threat Type, MITRE ATT&CK ID, Confidence Score (0-100%), Severity (Low/Medium/High/Critical), Human-readable Reason
Important: Ollama must be started with OLLAMA_ORIGINS="*" on the Windows host to allow cross-origin requests from the Ubuntu VM.

🗄️
InfluxDB — Time-Series Alert Database
DATABASE
▶
What It Is
Open-source time-series database optimized for storing and querying time-stamped data. Acts as the persistent memory and context store for the entire pipeline.
Deployment
Docker container, port 8086, on the Ubuntu SOC server. Version: InfluxDB 2.x. Queries written in Flux query language.
Key Uses
  • Alert history — 24-hour count and 5-minute velocity per source IP
  • Deduplication — prevents duplicate Telegram messages
  • Decision logging — stores every AI decision with full metadata
  • Isolated hosts tracking — records currently quarantined agents
Data Stored Per Alert
Timestamp, Source IP, Agent name, AI Decision, Threat type, Severity, Confidence score, MITRE ATT&CK ID
05
🚫
ip-blocker.py — Automated Firewall Response
CUSTOM API
▶
What It Is
A custom Python Flask API server written specifically for this project. Executes actual firewall blocking at the OS network level when n8n sends a BLOCK_IP decision.
Deployment
Runs as a systemd service on the Ubuntu SOC server, auto-starts on boot. Version: blocker.py v3.0. Listens on port 9999.
How It Works
  • Receives block/unblock requests from n8n via HTTP POST
  • Adds iptables INPUT chain DROP rules for attacking IP
  • Persists blocked IPs to /etc/iptables/blocked-ips for reboot survival
  • Has a /verify endpoint so n8n can confirm the block executed
Path
/home/soc/blocker.py on the SOC server. Blocked IPs persist at /etc/iptables/blocked-ips via netfilter-persistent.
06
📱
Telegram Bot — Real-Time Analyst Notification
NOTIFICATION
▶
What It Is
Telegram's Bot API used to deliver formatted security alert messages to the SOC analyst's phone in real time. Free, instant, works on any device with no additional infrastructure required.
Deployment
No server needed. n8n calls the Telegram Bot API directly via an HTTP Request node. A Telegram Bot token and chat ID are configured in the n8n workflow.
Message Contents
  • AI Decision (MONITOR / INVESTIGATE / BLOCK_IP / ISOLATE_HOST)
  • Threat Type and MITRE ATT&CK ID
  • Source IP address of the attacker
  • Affected agent / machine name
  • AI confidence score and severity level
  • Human-readable reason from the AI
  • Timestamp of the alert
Deduplication
InfluxDB is checked before sending. Duplicate alerts within the dedup window are suppressed to prevent notification spam to the analyst.
07
🌐
Tailscale — Secure Mesh Network Backbone
NETWORKING
▶
What It Is
A zero-configuration VPN built on WireGuard that creates a secure private mesh network between all devices. Without Tailscale, no component of the platform could communicate securely across machines.
Why It's Critical
  • Wazuh agents communicate with Wazuh manager over Tailscale
  • n8n calls the Ollama AI on Windows via Tailscale IP
  • No component is exposed to the public internet
  • Each machine gets a stable private 100.x.x.x IP that never changes
Deployment
Installed as a system service on all machines. Runs automatically on boot. No additional configuration needed after initial setup.
08
🔗
custom-webhook.py — Wazuh to n8n Bridge
INTEGRATION
▶
What It Is
A custom Python script that acts as the integration bridge between Wazuh and n8n. Wazuh's integratord module calls this script whenever an alert fires.
Why It's Needed
Wazuh cannot natively POST to arbitrary webhooks in the format n8n expects. This script handles the JSON translation and forwards the alert to the n8n webhook endpoint.
Deployment
Stored in a persistent Docker volume (wazuh_integrations). Auto-corrected by systemd service (wazuh-integrations-fix.service) on every boot to maintain correct file ownership (root:999).
Files
custom-webhook (no extension, the trigger script) and custom-webhook.py (the Python logic). Both must exist with correct permissions.
09
🔒
isolate-host.sh — Active Response Host Isolation
ACTIVE RESPONSE
▶
What It Is
A shell script registered as a Wazuh active response command. When the AI decides ISOLATE_HOST, this script runs directly on the affected agent machine and applies iptables rules that block ALL network traffic except the Wazuh manager connection.
Trigger Conditions
ISOLATE_HOST is triggered only for: Ransomware detection, Malware detection, C2 Beacon activity, and other severe threat classifications.
Commands Registered
isolate-host — applies network quarantine iptables rules.
unisolate-host — reverses isolation, restores full connectivity.
Tested With
Atomic Red Team ransomware simulation on LAB-SERVER. Wazuh detected the simulated ransomware behavior and ISOLATE_HOST triggered correctly, cutting off all network traffic.
Known Gap: Wazuh group config sync only pushes agent.conf, not arbitrary shell scripts. The isolation script must be manually placed on each agent. Future fix: Ansible-based automated deployment to all agents.

🤖
AI-Powered Agent Onboarding Assistant
WEB APP
▶
What It Is
A custom-built HTML/JavaScript web application served on port 8888 on the SOC server. Powered by Foundation-Sec-8B as the AI backend via Ollama. Guides analysts step-by-step through onboarding a new Wazuh agent onto the platform.
Key Features
  • AI chat interface for onboarding questions and commands
  • Smart context injection — feeds only relevant guide sections to AI
  • Covers full onboarding: Tailscale setup, Wazuh agent install, group assignment, script placement, verification
  • "Generate Update Report" modal — formatted session summary
  • Fully on-premise — no external API calls
Why It Matters
Reduces dependency on human expertise for onboarding. A non-technical analyst can onboard a new machine by following AI guidance without needing to know every command.
Deployment
Static HTML file with JavaScript. Version: soc_onboarding_v6.html. Backend AI calls go to Ollama on Windows host via Tailscale network.

Attacks Tested & Demonstrated

The following attack scenarios were executed against the live platform to verify end-to-end pipeline functionality. All attacks resulted in correct detection, AI classification, and automated response.

Tested Attack Scenarios
SSH BRUTE FORCE
SSH Brute Force Attack
Executed using Hydra from a TryHackMe AttackBox (IP: 100.x.x.x). Targeted the SSH service on the monitored Linux agent. Generated hundreds of failed login attempts within seconds.
✓ Detected by Wazuh Rule 5763 → AI classified as T1110 → IP auto-blocked via iptables within 3–6 seconds → Telegram alert sent
PORT SCAN
Network Port Scan
Executed using Nmap against the monitored agents. Full TCP SYN scan covering all common ports to simulate attacker reconnaissance phase.
✓ Detected by Wazuh Rule 40101 → AI classified as T1046 → INVESTIGATE / MONITOR decision → Telegram alert sent with MITRE mapping
FILE INTEGRITY
File Integrity Monitoring (FIM) Violation
Unauthorized file modifications performed inside monitored directories (/home/soc and /etc/iptables/). Simulates insider threat or post-compromise file tampering.
✓ Detected by Wazuh FIM engine → AI classified threat → Telegram alert fired with changed file details and timestamps
RANSOMWARE SIM
Ransomware Simulation — Host Isolation Test
Executed using Atomic Red Team framework on LAB-SERVER. Simulated ransomware behavior patterns including file encryption activity and C2-like behavior.
✓ Wazuh detected simulated ransomware behavior → AI classified as ISOLATE_HOST → isolate-host.sh ran on agent → All network traffic blocked except Wazuh manager → Host successfully quarantined

Future Scope

The following enhancements are planned for post-submission development and are acknowledged as known gaps in the current implementation.

Planned Future Enhancements
HIGH PRIORITY
SOC Dashboard — Real-Time Visual Monitoring
A dedicated web dashboard reading live data from InfluxDB to display total alerts, blocked IPs, severity breakdown, recent incident log, AI decision distribution, and currently isolated hosts. Currently not implemented — planned as next major feature.
HIGH PRIORITY
Ansible-Based Isolation Script Deployment
Automate deployment of isolate-host.sh to all agents via Ansible playbooks. Currently the script must be manually placed on each agent because Wazuh group config sync only pushes agent.conf, not arbitrary shell scripts.
MEDIUM
Block Verification Endpoint
A dedicated /verify endpoint on blocker.py plus an n8n confirmation node so the AI can re-verify and acknowledge whether the iptables block actually executed — not just report that it was requested.
MEDIUM
Windows & macOS Agent Response Support
Extend IP blocking and host isolation capabilities to Windows agents (using Windows Firewall / netsh) and macOS agents. Currently only Linux/Ubuntu agents support full automated response.
ROADMAP
TheHive Integration — Case Management
TheHive (open-source incident response platform) was partially deployed on port 9000 as a future case management layer. Full integration with n8n will enable analyst feedback loops and formal incident tracking.
ROADMAP
AI Intelligence Upgrade Layers
L2: Analyst feedback loop (requires TheHive + n8n wired). L3: AI-driven decisions using 2–3 months of L2 data. L4: Fine-tuning Foundation-Sec-8B on own incident data (6+ months of data required). Must follow this order strictly.
