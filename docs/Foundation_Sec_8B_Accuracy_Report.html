What the AI actually does in this pipeline

Foundation-Sec-8B acts as a threat labeller, not a decision-maker. The actual escalation decisions (monitor → investigate → block → isolate) are made by the deterministic Decision Engine based on effectiveCount thresholds. The AI contributes:

✓ threat_type label ✓ MITRE ATT&CK ID ✓ Human-readable reason ✓ Confidence score ~ action (advisory only) ✗ Final decision (Decision Engine owns this)

Prompt Engineering Journey

The system prompt evolved through multiple iterations during testing. Each iteration addressed observed failures from the previous round.

VERSION 1 — Original prompt (v12.9.9)
Generic SOC analyst instruction. No count reasoning. No confidence rules. Result: AI classified every alert as ssh_brute_force regardless of context. All executions identical — prompt_eval_count always 334.
ROOT CAUSE FOUND — Template variables not rendering
The {{ }} expressions inside the HTTP Request body were being sent to Ollama as literal text. Ollama received "{{ $json.rule.description }}" as the alert — not the actual value. Fixed by adding Build Ollama Prompt Code node that resolves all values in JavaScript before the HTTP call.
VERSION 2 — Count-aware prompt
Added reasoning rules: count 0-2 = isolated event, 3-9 = pattern, 10+ = confirmed. Confidence capped at 0.5 for first alerts. Action rules strictly tied to count and velocity. Result: prompt_eval_count jumped from 334 to 645 — AI reading full prompt.
VERSION 3 — Gemini malware override added
Added Rule 6: CRITICAL OVERRIDE FOR MALWARE — if alert describes process injection, malware, rootkits, ACTION MUST BE isolate_host immediately even at count 1. Tested on process injection scenario — AI correctly returned isolate_host. This was the missing rule that caused Test 5 failure in earlier rounds.
FINAL — If node threshold fixed
Discovered the If node had condition (rule.level >= 7) — all level 6 alerts (port scans, low severity) were routed to Normalize Localhost Data, completely bypassing Ollama. Changed to >= 5. This explained why the AI "always said ssh_brute_force" — only SSH level 8+ alerts were ever reaching it.
Final System Prompt
You are a SOC Analyst API for an enterprise network. Analyze the security alert and respond with JSON only. Before classifying, reason through these questions: 1. How many times has this IP appeared in the last 24 hours? - 0 to 2 times: isolated event, low confidence, do not escalate - 3 to 9 times: pattern emerging, moderate confidence - 10 or more: repeated behavior, high confidence 2. What is the velocity in last 5 minutes? - 0 to 2: could be human error or normal failure - 3 to 9: suspicious, elevated - 10 or more: automated, very high confidence 3. What was the last SOC decision for this IP? - none → first time seen, be cautious - monitor → re-appearing IP, raise severity slightly - investigate → already suspicious, escalate - block_ip → was blocked before, treat seriously 4. Confidence rules (strict): - count 0-2: max 0.5 · count 3-9: 0.5-0.75 · count 10+: 0.75-0.95 5. ACTION strict rule: - count 0-2 AND velocity 0-2 → monitor - count 3-9 OR velocity 3-9 → investigate - count 10+ OR velocity 10+ → block_ip 6. CRITICAL OVERRIDE FOR MALWARE: If the alert describes process injection, malware, rootkits, or critical host compromise → ACTION MUST BE isolate_host immediately. 7. threat_type must reflect COUNT not just alert text 8. Reason field must reflect actual numbers received

Key Findings

🔴 SSH Bias — Model Training Issue
Foundation-Sec-8B classifies SSH-related alert text as ssh_brute_force regardless of count. This is baked into model weights from training data. Not fixable with prompt engineering alone. Affects tests 1, 6, 13. Action is still correct — this is cosmetic for the analyst.
🔴 T1110 as Default MITRE
When the model is uncertain about MITRE mapping, it defaults to T1110 (Brute Force). Affected port scan (T1046) and lateral movement (T1021). Fixed by adding a deterministic MITRE_MAP in the Decision Engine — zero speed impact, 100% accuracy guaranteed.
🟡 If Node Level Threshold Bug
The If node had condition rule.level >= 7 — all level 5-6 alerts (port scans, recon) were bypassing Ollama entirely via the localhost branch. This explained the two-week "AI always says SSH brute force" mystery. Fixed by changing threshold to >= 5.
🟡 Confidence Too High on Malware
Test 4 (process injection at count 0) returned confidence 0.95. At count 0, confidence should be capped at 0.5 per our rules. However, the malware override rule correctly fired isolate_host, so functionally the output was correct. Minor rule inconsistency.
🟢 Reason Field — Excellent Quality
All 14 tests produced accurate, context-aware reason fields that correctly reflected actual count and velocity numbers. Phrases like "appeared 0 times... first-time event with low confidence" show genuine contextual awareness. This is the AI's strongest contribution.
🟢 Malware Override Working
Rule 6 (CRITICAL OVERRIDE FOR MALWARE) correctly triggered isolate_host on process injection (Test 4) even at count 0. Previously the AI would return monitor or investigate for malware alerts. This was the most important prompt improvement.

Final Assessment
74% raw accuracy is a solid result for an 8B parameter security-specialized model with no fine-tuning on this environment.

The AI performs best on non-SSH threats — data exfiltration, SQL injection, privilege escalation, reconnaissance, and malware all classified correctly. The primary weakness is SSH-related alerts where model training bias causes brute_force labelling even at count 0.

Key architectural insight: The AI's role in this pipeline is correctly scoped. It provides threat labels and human-readable analyst context. The actual escalation decisions are made deterministically by the Decision Engine. This design means AI accuracy failures are mitigated — a wrong threat_type label does not cause a wrong block or isolate action.

The effective pipeline accuracy is approximately 90% when accounting for Decision Engine overrides that catch AI failures on malware and high-severity alerts.

The remaining 10% gap — SSH classification bias and rare action mismatches — represents the fundamental limitation of 8B parameter models and would require either fine-tuning on environment-specific data (L4 roadmap) or a larger model class to fully close.
