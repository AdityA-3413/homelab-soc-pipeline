01

Network Attacks

01 🔨 SSH Brute Force ✓ FULLY DETECTED
▶
What it is
Attacker tries thousands of username/password combinations on SSH port 22 to gain access. Like trying every key on a keyring until one works.
How we detect it
Wazuh rule 5716 fires on each failed login. n8n counts them — at 15+ triggers auto-block. iptables blocks the IP. Telegram confirmed message sent.
Pipeline response
MONITOR → INVESTIGATE → BLOCK_IP → ✅ IP BLOCKED CONFIRMED. Tested with real Hydra attack. Fully verified.
02 🔍 Port Scanning / Network Reconnaissance ✓ FULLY DETECTED
▶
What it is
Attacker scans all ports to find open services — like knocking on every door in a building to see which ones open. Usually the first step before any real attack.
How we detect it
Wazuh rule 1002 detects multiple connection attempts to different ports. AI classifies as port_scan, MITRE T1046. Pipeline escalates based on velocity.
Pipeline response
Low velocity → MONITOR. High velocity (Nmap aggressive scan) → INVESTIGATE → eventually BLOCK_IP. Works after If node threshold fix (>= 5).
03 📁 FTP Brute Force ✓ FULLY DETECTED
▶
What it is
Same as SSH brute force but targeting FTP file transfer port. Attacker tries to get into the file server to steal or modify files.
How we detect it
Wazuh has built-in FTP authentication failure rules. Same pipeline — failed logins counted, IP blocked after threshold reached.
Pipeline response
Same MONITOR → BLOCK_IP escalation as SSH. MITRE T1110. Needs FTP service running on the monitored machine.
04 💉 SQL Injection ⚠ PARTIAL
▶
What it is
Attacker injects malicious SQL code into a web form to steal or delete database data. Like tricking a form into running commands instead of storing data.
How we detect it
Wazuh rule 31103 detects SQL injection patterns in web logs. AI correctly classified as sql_injection_attempt, MITRE T1190 in testing.
Gap
Only detects if a web application is running on the monitored server AND web server logs are configured in Wazuh. No web app currently running on SIEM-SERVER.
To improve: Deploy a web application on the monitored server, configure Apache/Nginx log monitoring in Wazuh ossec.conf.
05 💥 DDoS — Distributed Denial of Service ✗ NOT DETECTED
▶
What it is
Thousands of machines flood your server with requests simultaneously to make it crash. Like 10,000 people calling the same phone number at once — line goes dead.
Why not detected
Tailscale blocks external traffic completely. A DDoS from the public internet cannot reach your server. No external IP can connect. This attack is architecturally impossible on your network.
Real risk
If Tailscale itself goes down or gets DDoS'd, your network connectivity breaks. But that's Tailscale's problem to handle, not yours.
Note: Not detected but also not possible on your architecture. This is a non-issue for your threat model.
02

Endpoint / Host Attacks

06 💀 Malware / Process Injection ✓ FULLY DETECTED
▶
What it is
Malicious code injected into a running process — attacker hides inside a legitimate program like svchost.exe. The malware runs under a trusted process so security tools miss it.
How we detect it
Wazuh rule 87105 detects suspicious child processes and memory injection. AI receives it → Rule 6 malware override fires → isolate_host immediately regardless of count.
Pipeline response
ISOLATE_HOST triggered. blocker.py sets iptables policy to DROP all. Only Wazuh manager stays connected. Telegram 🔒 HOST ISOLATED. Tested and verified.
07 🔐 Ransomware ⚠ PARTIAL
▶
What it is
Malware that encrypts all your files and demands payment to decrypt them. Attacker locks your data and holds it hostage. Most damaging attack for businesses.
How we detect it
Wazuh rule 62300 detects mass file encryption patterns. AI classifies as file_encryption, MITRE T1486. Decision Engine rule-ID override triggers ISOLATE_HOST via rule 62300.
Gap
Detection happens AFTER encryption starts. By the time Wazuh sees it, some files are already encrypted. No pre-emptive detection. No backup verification in pipeline.
To improve: Add file integrity monitoring (FIM) in Wazuh for critical directories. Alert on bulk file modification BEFORE encryption completes. Integrate backup verification.
08 ⬆️ Privilege Escalation ✓ FULLY DETECTED
▶
What it is
Attacker gains higher permissions than they should have. Like a regular employee accessing the CEO's files. They exploit misconfigurations or bugs to become admin/root.
How we detect it
Wazuh rule 5402 detects unauthorized sudo usage. AI correctly classifies as unauthorized_sudo_usage, MITRE T1548.003. Pipeline escalates appropriately.
Pipeline response
Depends on count. First occurrence → MONITOR. Repeated → INVESTIGATE → analyst reviews in TheHive. Tested in accuracy evaluation — correct classification.
09 🕳️ Rootkit Detection ✓ FULLY DETECTED
▶
What it is
Deep-level malware that hides itself from the operating system. Like a burglar who also hides the security camera footage. Extremely dangerous — hard to find.
How we detect it
Wazuh rules 510, 511, 533 — built-in rootkit detection scans. Decision Engine ISOLATE_RULE_IDS includes all three. Immediate isolate_host regardless of count.
Pipeline response
Immediate ISOLATE_HOST. No count threshold needed. Rule ID match is authoritative. Host isolated, Wazuh manager stays connected for forensics.
10 🔗 Persistence Mechanisms ⚠ PARTIAL
▶
What it is
Attacker installs backdoors so they survive reboots. Creating scheduled tasks, adding startup scripts, creating new admin users, adding SSH keys. They want to come back even after you restart.
How we detect it
Wazuh detects: new user creation (rule 5501), cron job changes, SSH authorized_keys modification. Linux persistence is covered. Windows persistence has gaps.
Gap
Windows registry run keys, scheduled tasks, WMI persistence — not detected without Windows agent on employee machines. Windows persistence is completely blind currently.
To improve: Install Wazuh agent on Windows machines. Enable Windows registry monitoring and scheduled task monitoring in Wazuh configuration.
03

Insider & Social Engineering Threats

11 👤 Insider Threat — Authorized User Doing Malicious Things ✓ FULLY DETECTED
▶
What it is
Employee intentionally or accidentally causes damage — accessing files they shouldn't, exporting data, sabotaging systems. The attacker already has legitimate access.
How we detect it
Wazuh detects access outside business hours, sensitive file access, unusual commands. AI recognizes Tailscale IPs and treats them cautiously — uses count and velocity to escalate.
Pipeline response
AI correctly returns investigate action. Tested with john.smith accessing files outside hours — correct classification, MITRE T1078. Escalates to analyst in TheHive.
12 🎣 Phishing — Employee Clicks Malicious Link/Attachment ✗ NOT DETECTED
▶
What it is
Attacker sends a fake email pretending to be a trusted source. Employee clicks a link or opens an attachment. Malware installs silently. This is how 80-90% of enterprise breaches start.
Why not detected
Phishing happens at email/browser level on employee machines. Wazuh on the Linux server cannot see what happens on employee laptops. No endpoint agent on employee Windows machines currently.
What we CAN detect
The AFTERMATH of phishing — when malware starts acting: lateral movement, suspicious processes, C2 communication. We see it after execution, not at click time.
To improve: Install Wazuh agent on all employee Windows machines. Wazuh detects: Office macros spawning PowerShell, browser spawning unexpected child processes, encoded PowerShell execution.
13 💻 Hijacked Employee Device — Attacker Operating Inside Tailscale ↑ IMPROVABLE
▶
What it is
The most realistic threat to your architecture. Attacker compromises an employee laptop via phishing. Now they operate from INSIDE Tailscale — they can reach every machine on the network.
Currently detected
When the attacker moves laterally — SSH attempts from internal Tailscale IP to SIEM-SERVER are detected. AI handles Tailscale IPs correctly. Escalates based on count.
Gap
Everything happening ON the compromised employee machine is invisible. Credential dumping, recon, C2 communication — all blind until attacker moves to another machine.
To improve: Wazuh agent on WORKSTATION-01. This single change gives you visibility into the most likely real attack scenario on your network.
14 🗣️ Social Engineering — Vishing, Pretexting ✗ NOT DETECTED
▶
What it is
Attacker calls an employee pretending to be IT support, tricks them into giving passwords or installing software. Pure human manipulation — no technical exploit needed.
Why not detected
This is a human problem, not a technical one. No SIEM in the world can detect a phone call. The detection only happens after the attacker uses the stolen credentials.
What we detect after
When stolen credentials are used — login from unusual IP, access at unusual time, accessing sensitive files. AI detects the behaviour, not the method.
Note: No technical fix possible. Requires employee security awareness training. Outside the scope of a SIEM/SOC tool.
04

Advanced & Sophisticated Attacks

15 ↔️ Lateral Movement — Spreading Across Network ⚠ PARTIAL
▶
What it is
Once inside one machine, attacker moves to other machines on the network. Like a burglar who breaks into one apartment and then tries every other door in the building.
How we detect it
SSH attempts from internal Tailscale IPs to SIEM-SERVER are detected. Wazuh sees the connection attempts. AI classifies the pattern. MITRE T1021 via MITRE map fix.
Gap
Lateral movement between employee machines (WORKSTATION-01 → LOQ → SERVER-01) is completely invisible unless those machines have Wazuh agents. We only see it when it hits the Linux server.
To improve: Wazuh agents on all Tailscale machines. Then lateral movement between any two machines is visible and creates alerts.
16 📤 Data Exfiltration — Stealing Company Data ⚠ PARTIAL
▶
What it is
Attacker copies company data to external servers. Employee records, financial data, source code, passwords. The theft is silent — files are copied, not deleted.
How we detect it
Wazuh rule 87200 detects large outbound data transfers. AI correctly classified as data_exfiltration_attempt, MITRE T1041 in testing. Pipeline escalates to investigate.
Gap
DNS tunneling exfiltration is invisible — attacker hides data in DNS queries which look like normal traffic. Also misses exfiltration from employee machines without agents.
To improve: Enable Wazuh network monitoring rules. Add DNS query monitoring. Set baseline for normal outbound traffic volume and alert on deviations.
17 📡 C2 — Command and Control Communication ✓ FULLY DETECTED
▶
What it is
Malware on a machine calls home to the attacker's server to receive instructions. Like a spy reporting back to headquarters. The attacker remotely controls the infected machine.
How we detect it
Wazuh rules 87106, 87110 detect C2 beaconing patterns. Decision Engine ISOLATE_RULE_IDS covers both. AI secondary check also catches c2_beacon threat_type. ISOLATE_HOST fires immediately.
Pipeline response
Immediate ISOLATE_HOST. Machine cut off from network. Only Wazuh manager stays connected. Blocks the attacker from continuing to control the machine.
18 🔑 Credential Dumping — Stealing Passwords from Memory ↑ IMPROVABLE
▶
What it is
Attacker runs tools like Mimikatz to extract passwords stored in Windows memory. Gets all cached credentials in seconds — email passwords, domain passwords, everything.
Currently
ISOLATE_DESC_KEYWORDS includes 'mimikatz' and 'credential dump'. If Wazuh detects these keywords in a rule description, ISOLATE_HOST fires immediately.
Gap
Wazuh on Linux server cannot see Mimikatz running on a Windows employee machine without a Windows agent installed there.
To improve: Wazuh Windows agent + Sysmon integration. Sysmon logs every process execution — Mimikatz activity is immediately visible.
19 0️⃣ Zero-Day Exploits — Unknown Vulnerabilities ✗ NOT DETECTED
▶
What it is
Attacker exploits a software vulnerability that nobody knows about yet — no patch exists. The most sophisticated attacks use zero-days. Nation-state level attack capability.
Why not detected
By definition, no signature exists for an unknown vulnerability. Rule-based detection systems cannot detect what they don't know about. Even enterprise SOCs struggle with this.
What we detect after
The BEHAVIOUR after exploitation — privilege escalation, lateral movement, persistence, C2 — all of these are detected by your pipeline even if the initial exploit method is unknown.
Note: No rule-based system detects zero-days. Requires ML-based anomaly detection (your L3 roadmap). Current detection of post-exploitation behaviour is the best realistic defence.
20 ⛓️ Supply Chain Attack — Compromised Software Update ✗ NOT DETECTED
▶
What it is
Attacker compromises a software vendor's update system. When employees update their software, they unknowingly install malware. SolarWinds attack was this — affected 18,000 companies.
Why not detected
The malware arrives as a legitimate signed software update. Looks completely normal. No brute force, no scanning, no obvious indicators. Extremely sophisticated.
What we detect after
Once the malware activates — C2 beaconing, lateral movement, data exfiltration — your pipeline detects the resulting behaviour. The initial infection method is invisible.
Note: Beyond the scope of a SIEM to prevent. Requires software composition analysis and vendor risk management at organisational level.
